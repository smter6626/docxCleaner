# docxCleaner 静态规格说明

## 1. 文档定位

本文件是 `docxCleaner` 项目的长期目标与稳定约束说明，用于记录项目的大方向、范围、非目标、开发边界和验收标准。

本文件只在以下情况更新：

- 用户调整项目目标；
- 用户调整核心功能范围；
- 用户调整目标平台、技术栈或交付形式；
- 用户明确要求修改长期约束。

普通开发进度、阶段完成情况、当前任务和临时问题不写入本文件，应写入 `runtime.md`。

---

## 2. 项目目标

`docxCleaner` 的目标是把现有的 DOCX 元数据查看脚本升级为一个面向普通用户的图形化工具，用于查看、理解、编辑和整理 DOCX 类文件中的源数据与元数据。

项目的长期目标不是只做一个简单的“时间戳修改器”，而是逐步发展为一个 UI 友好、人机交互简单、可维护、可扩展的 DOCX 元数据查看与清理工具。

核心方向：

1. 尽可能多地读取 DOCX 类文件中的元数据与可疑源数据；
2. 尽可能多地提供可控、透明、可验证的编辑能力；
3. 用 GUI 替代命令行交互，降低使用门槛；
4. 优先保证修改后的文件仍能被 Microsoft Word 正常打开；
5. 优先保证用户知道每项修改会影响什么，而不是盲目“一键伪装”。

---

## 3. 使用边界

本项目用于个人文档发布前的隐私检查、元数据整理和文档属性管理。

本项目不以伪造作者身份、篡改证据链、规避学术诚信审查、规避平台检测或隐藏不当行为为目标。

涉及“清理”“替换”“重写”等功能时，文案应尽量使用中性描述，例如：

- 元数据整理；
- 隐私字段清理；
- 文档属性重写；
- 发布前检查；
- 可疑源数据提示。

避免把项目描述成“反检测工具”或“伪装工具”。

---

## 4. 目标用户

目标用户是需要在本地查看或整理 DOCX 文件元数据的普通用户，包括：

- 不熟悉命令行的用户；
- 需要发布 DOCX 文件前检查隐私字段的用户；
- 需要理解 DOCX 内部元数据来源的用户；
- 需要批量或半自动整理文档属性的用户。

用户不应被要求理解 WordprocessingML、ZIP 包结构或 XML 命名空间，除非进入高级模式。

---

## 5. 目标文件类型

第一阶段目标文件类型：

- `.docx`

长期可扩展目标文件类型：

- `.docm`
- `.dotx`
- `.dotm`
- 其他基于 Office Open XML 结构的 Word 文档类型

当前阶段不要过早扩展到 PPTX 或 XLSX。仓库中如存在 `PPTwhereisAI.py`，暂时视为历史相关文件或未来参考，不作为当前主线目标。

---

## 6. 技术栈约束

主要语言：

- Python 3

GUI 技术：

- 优先使用 Python 标准库 `tkinter`。

依赖策略：

- 默认尽量不依赖第三方库；
- `Pillow` 可作为可选依赖，用于图片 EXIF 检查；
- 如果 `Pillow` 未安装，程序应显示“未安装 Pillow，跳过 EXIF 检查”，不得崩溃；
- 不强依赖 `tkcalendar`；日期时间选择器优先用 `Spinbox` 组合实现。

文件处理策略：

- DOCX 按 ZIP 包读取；
- XML 使用 Python 标准库 `xml.etree.ElementTree` 或必要时使用字符串级最小修改；
- 修改 DOCX 时应尽量只改目标 XML 文件；
- 未修改的 ZIP 条目应尽量保留原始内容、路径、压缩方式和外部属性。

---

## 7. 平台目标

主要开发环境：

- macOS

长期目标平台：

- macOS
- Windows

优先级：

1. 先保证 macOS 上可运行、可分析、可生成有效 `_cleaned.docx`；
2. 再处理 Windows 兼容性；
3. 如果跨平台实现与 macOS 稳定性冲突，优先保证 macOS 主流程可用。

跨平台注意事项：

- 打开文件夹功能需要区分 macOS、Windows 和 Linux；
- 路径处理必须使用 `pathlib.Path`；
- 不硬编码 `/Users/...` 作为程序内部路径；
- GUI 字体和窗口尺寸应避免依赖单一系统。

---

## 8. 核心功能范围

### 8.1 查看功能

程序应能读取并展示 DOCX 中的主要元数据与源数据，包括但不限于：

- `docProps/core.xml` 核心属性；
- `docProps/app.xml` 应用属性；
- `docProps/custom.xml` 自定义属性；
- `word/settings.xml` 中的 RSID 信息；
- `word/document.xml` 中的正文 RSID 属性；
- 修订记录；
- 批注；
- 样式一致性；
- 语言分布；
- 字数统计；
- ZIP 内部文件时间戳；
- 内嵌媒体文件与图片 EXIF 信息。

查看功能必须以 GUI 报告形式呈现，不应默认写入本地报告文件。

### 8.2 编辑功能

程序应逐步支持编辑或整理以下信息：

- 创建者；
- 最后修改人；
- 创建时间；
- 修改时间；
- 编辑时长；
- ZIP 内部文件时间戳；
- RSID 列表；
- 正文中的 RSID 属性；
- 未来可扩展到批注、修订记录、自定义属性、语言标记、内嵌媒体 EXIF 等。

编辑必须遵循可验证原则：修改后应能再次分析生成的新文件，并在报告中看到字段变化。

### 8.3 一键整理功能

程序可以提供“一键清理”或“一键整理”功能，但必须满足：

- 默认值明确；
- 修改范围明确；
- 输出文件不覆盖原文件；
- 生成文件后用户可再次分析验证；
- 不应把不可逆修改隐藏在模糊按钮后。

输出文件命名规则：

- 原文件名 stem + `_cleaned.docx`；
- 保存到原文件相同目录；
- 不覆盖原始文件。

---

## 9. GUI 设计原则

GUI 应遵循以下原则：

1. 主流程简单：选择文件 → 分析 → 查看报告 → 修改或一键整理；
2. 报告区域清晰，能复制文本；
3. 修改窗口字段明确，避免用户自由输入不合法日期；
4. 错误信息用普通人能理解的语言展示；
5. 不把高级 XML 概念暴露给普通用户，除非作为高级信息展示；
6. 操作完成后明确显示输出路径；
7. 任何修改都不应静默覆盖原文件。

## 10. 当前非目标

当前阶段不做以下内容：

- 不写 README；
- 不做安装包；
- 不做批量处理；
- 不做云端上传；
- 不做 Web 服务；
- 不做复杂主题皮肤；
- 不优先支持 PPTX/XLSX；
- 不承诺彻底删除所有可追踪信息；
- 不承诺生成的文件能绕过任何平台、学校或机构的检测。

## 11. 开发方式

本项目采用双文档工作流：

- `static.md`：长期稳定规格；
- `runtime.md`：动态执行状态。

开发时应优先更新 `runtime.md`，除非目标或硬约束变化。

每个 active 任务必须满足：

- 目标明确；
- 输入文件明确；
- 操作步骤明确；
- 验收方式明确；
- 缺失上下文必须显式标记。

---

## 12. 验收标准

阶段性最低验收标准：

1. 程序能在 macOS 上启动 GUI；
2. 用户能选择 `.docx` 文件；
3. 程序能在 GUI 中显示完整分析报告；
4. 程序能生成 `_cleaned.docx`，且不覆盖原文件；
5. 生成文件能被 Microsoft Word 正常打开；
6. 生成文件能被本工具再次分析；
7. 修改过的字段能在再次分析中被验证；
8. 文件损坏、XML 缺失、Pillow 缺失等情况不会导致程序崩溃。

长期验收标准：

1. 普通用户不需要命令行即可完成主要操作；
2. 主要元数据字段可以查看、理解、修改和验证；
3. macOS 与 Windows 均可运行主流程；
4. 代码结构可维护，功能扩展不需要重写主程序。

---

## 13. 重新规划规则

出现以下情况时，需要重新规划并可能更新本文件：

- 用户决定从 DOCX 扩展到 Office 全家桶；
- 用户决定改用 PyQt、Electron 或 Web GUI；
- 用户决定优先做 Windows；
- 用户决定加入批量处理；
- 用户决定把项目定位为正式发布工具；
- 用户明确改变项目用途或使用边界。

否则，普通开发进展只更新 `runtime.md`。
